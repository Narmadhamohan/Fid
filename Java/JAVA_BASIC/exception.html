Builtin Exceptions

Builtin Exceptions in Java

Java provides several built-in exceptions that are automatically thrown during runtime when certain conditions are met. These are part of the standard Java library.

  • NullPointerException: Thrown when trying to use a `null` object reference.
  • ArrayIndexOutOfBoundsException: Thrown when accessing an invalid array index.
  • ArithmeticException: Thrown during arithmetic errors, like division by zero.
Example: 

public class BuiltInExceptionExample {
    public static void main(String[] args) {
        try {
            int[] arr = new int[5];
            System.out.println(arr[10]); // This will throw ArrayIndexOutOfBoundsException
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Exception caught: " + e);
        }
    }
}
Handling Exceptions

Handling Exceptions in Java

Handling exceptions in Java ensures that runtime errors are dealt with smoothly, preventing abrupt program termination.

  • try: Contains code that might throw exceptions.
  • catch: Catches and handles the exceptions thrown by the `try` block.
  • finally: Block that executes regardless of whether an exception was thrown or caught. Often used for cleanup.
Example: 

public class ExceptionHandlingExample {
    public static void main(String[] args) {
        try {
            int result = 10 / 0; // This will throw ArithmeticException
        } catch (ArithmeticException e) {
            System.out.println("Cannot divide by zero!");
        } finally {
            System.out.println("This will always be executed.");
        }
    }
}
Custom Exceptions

Creating Custom Exceptions in Java

You can define your own exception classes to represent specific error conditions in your program by extending either the `Exception` class or the `RuntimeException` class.

  • Checked exceptions: Must be declared in method signatures using `throws` keyword and are checked during compile-time.
  • Unchecked exceptions: Do not need to be declared in method signatures and are checked at runtime.
Example: 

class CustomException extends Exception {
    public CustomException(String message) {
        super(message);
    }
}

public class CustomExceptionExample {
    public static void main(String[] args) {
        try {
            checkAge(15);
        } catch (CustomException e) {
            System.out.println("Exception caught: " + e.getMessage());
        }
    }

    static void checkAge(int age) throws CustomException {
        if (age < 18) {
            throw new CustomException("Age must be 18 or older to vote.");
        } else {
            System.out.println("You are eligible to vote.");
        }
    }
}
Throwing Exceptions

Throwing Exceptions in Java

The `throw` keyword in Java allows you to explicitly throw an exception, signaling an error condition during the program's execution.

  • Use `throw` followed by an instance of the `Exception` class or one of its subclasses.
  • This is often used to throw user-defined or unchecked exceptions.
Example: 

public class ThrowingExceptionExample {
    public static void main(String[] args) {
        try {
            validate(0);
        } catch (IllegalArgumentException e) {
            System.out.println("Exception caught: " + e);
        }
    }

    static void validate(int number) {
        if (number <= 0) {
            throw new IllegalArgumentException("Number must be greater than zero.");
        } else {
            System.out.println("Valid number: " + number);
        }
    }
}
Exception Hierarchy

Java Exception Hierarchy

The Java exception hierarchy starts from the root class `Throwable`, which has two main branches: `Error` and `Exception`. All exceptions in Java are derived from these classes.

  • Throwable: The superclass of all errors and exceptions.
  • Error: Represents serious issues that are not typically handled by applications (e.g., `OutOfMemoryError`).
  • Exception: Represents conditions that applications might want to catch (e.g., `IOException`).
  • RuntimeException: A subclass of `Exception` that represents exceptions that can occur during the runtime (e.g., `NullPointerException`).
